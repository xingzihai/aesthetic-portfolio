# 涟漪系统坐标分析

## 问题现象

- 背景变黑色
- 元素位置有偏差

## 根本原因：坐标系统不匹配

### 三个坐标系统

| 系统 | 原点 | Y轴方向 | 当前用途 |
|------|------|---------|----------|
| **屏幕坐标** | 视口左上角 | 向下增长 | 点击位置 `e.clientX, e.clientY` |
| **WebGL坐标** | Canvas左下角 | 向上增长 | Shader渲染，`glY = canvasHeight - y` |
| **纹理坐标** | 纹理左下角 | 向上增长 | 采样，`(0,0)` 到 `(1,1)` |

### 关键问题

当捕获 `.surface-world` 时：

```
纹理内容 = surface-world 从 scrollTop 开始的 canvasHeight 高度区域
```

但这个区域的内容 ≠ 当前视口显示的内容！

因为 `surface-world` 是 `position: relative`，它从文档顶部开始。
- 当 `scrollTop = 500` 时，视口显示的是 surface-world 的 500-500+canvasHeight 区域
- 但 html2canvas 的 `y: scrollTop` 参数是从元素自身的 scrollTop 位置开始捕获
- 对于 `position: relative` 元素，它的 scrollTop 始终是 0（因为元素本身不滚动）

**结论**：`y: scrollTop` 对 `.surface-world` 无效！

### 纹理与视口的关系

```
纹理[0, 1] 范围对应 surface-world[scrollTop, scrollTop+canvasHeight]
视口显示的内容是 surface-world[scrollTop, scrollTop+canvasHeight]（相对于文档）

但 Shader 中：
screenPos = v_texCoord * resolution  （视口坐标）
涟漪中心 = 点击位置（视口坐标）

纹理采样时：
texUV = distortedUV （假设纹理[0,1] = 视口[0, canvasHeight]）

这假设了纹理内容与视口内容一一对应！
```

**但捕获 `.surface-world` 时，这个假设不成立**，因为：
- 纹理是 surface-world 的"裁剪视图"
- 视口是整个文档的"滚动视图"
- 两者坐标系不同

## 解决方案

### 方案A：恢复捕获 document.body

这是原版方案，保证纹理与视口一一对应。

优点：
- 坐标系统一致
- 背景、元素位置正确

缺点：
- 会捕获背后世界内容（但涟漪不会实际扭曲它）

### 方案B：正确捕获 surface-world

需要计算 texture offset：

1. 捕获整个 surface-world（不裁剪）
2. 在 Shader 中计算纹理偏移：

```glsl
// 视口内容在纹理中的位置
float textureOffsetY = scrollTop / surfaceWorldHeight;
texUV.y = texUV.y + textureOffsetY;
```

但这需要知道 surface-world 的完整高度，且纹理可能很大。

### 方案C：使用 getElementRelativePosition

计算 surface-world 相对于视口的偏移，然后调整捕获参数。

但这很复杂，且 html2canvas 的参数行为难以控制。

---

## 推荐：方案A

恢复捕获 `document.body`，设置正确的背景色。

**为什么这满足用户需求**：

用户说"涟漪不影响背后世界"——这指的是**视觉效果上不影响**。

当前涟漪系统：
- 涟漪 Canvas 是 `z-index: 100`，在所有内容之上
- 背后世界是 `z-index: 1`，在最底层
- 涟漪只是显示扭曲的纹理，不会实际改变背后世界的渲染

所以捕获 `document.body` 时：
- 纹理包含背后世界
- 涟漪显示扭曲效果
- 但背后世界本身不受影响（它在涟漪层下方）

唯一的视觉问题是：涟漪扭曲区域内会看到"扭曲的背景世界"，但这是涟漪纹理的一部分，不是真实的背后世界。

---

## 执行方案A

1. 恢复捕获 `document.body`
2. 设置 `backgroundColor: '#E8DFD6'`（页面背景色）
3. 测试效果
